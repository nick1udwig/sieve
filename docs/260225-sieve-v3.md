## sieve v3

- Event-loop
- Concurrent loops
- Fast rust
- mpsc & tokio for tasks
- fide-style plan-and-quarantine split with strings into planner only allowed from user intent: quarantine is only allowed to output enums, ints, floats, bools -- simple and predefined types so that it cannot prompt inject the planner
- plugin user interfaces; start with telegram long polling, but expand to other chat interfaces (and perhaps even custom ones in future)
- control computer using exclusively bash commands
  - implies control policy is entirely in terms of what bash commands can do, e.g. `ls` reads locally; `>` writes locally; `curl -X GET` reads remotely; `curl -X POST` writes remotely; etc

- browser use through agent-browser cli becomes just a scriptable thing -> becomes cli (post-MVP)

- main problem becomes:
  - how to programatically determine the caps required for a cli call
  - e.g. say we assign some by hand (basic commands like `ls` `cat` etc etc)
  - can we derive the caps of some composition?
    e.g. a bash script
         or `ls | grep 'hi' > foo.txt`

- we can use a lot of work done in codex cli here, e.g. ~/git/codex/codex-rs/linux-sandbox for bwrap sandboxing to safely run bash commands and ~/git/codex/codex-rs/shell-command which defines safe and unsafe commands as well as parsing on bash
- we care only about linux
- our caps are quite different from codex's: we want to maintain ongoing information about info provenance so we can enforce what can be done with that info subsequently

- we can run unknown commands in bwrap to avoid allowing ill effects to propagate
- we can further run unknown commands in `strace -ff` so we can learn about what network, disk, etc the command is trying to use. we can use this to inform user that, for this exact set of args, this command has these caps
